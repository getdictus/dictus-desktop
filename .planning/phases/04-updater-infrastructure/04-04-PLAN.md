<!-- UPDT-10 success criterion — non-negotiable -->

Target endpoint: `https://github.com/getdictus/dictus-desktop/releases/latest/download/latest.json`

Assertion command (verbatim):
```bash
curl -sfL https://github.com/getdictus/dictus-desktop/releases/latest/download/latest.json \
  | jq -e '.version and .platforms and (.platforms | keys | length > 0)'
```
Must exit 0 and the fetched JSON must have:
- `.version` — non-empty string (expected: `"0.1.0"`)
- `.platforms` — object with at least 1 key (expected: all 7 from the build matrix)
- Each platform key → `.signature` (non-empty) and `.url` (non-empty)

Expected platform keys (per tauri-action mapping of the build matrix in `release.yml:47-69`):
- `darwin-aarch64` (macos-26, aarch64-apple-darwin)
- `darwin-x86_64` (macos-latest, x86_64-apple-darwin)
- `linux-x86_64` (ubuntu-22.04 or ubuntu-24.04, x86_64-unknown-linux-gnu)
- `linux-aarch64` (ubuntu-24.04-arm, aarch64-unknown-linux-gnu)
- `windows-x86_64` (windows-latest, x86_64-pc-windows-msvc)
- `windows-aarch64` (windows-11-arm, aarch64-pc-windows-msvc)

Note: the ubuntu-22.04 and ubuntu-24.04 jobs both target `x86_64-unknown-linux-gnu` but produce different bundle types (deb vs appimage+rpm). tauri-action may publish only one of them to `latest.json` as `linux-x86_64`. Either is acceptable — the assertion only requires `.platforms | keys | length > 0`, not a specific 6+ platform coverage. Per 04-RESEARCH.md Open Question §3, verify the actual platforms object during Task 2 and document the result in the SUMMARY.

Pre-flight command (per 04-CONTEXT.md, must run BEFORE triggering release.yml):
```bash
git tag -d 0.1.0 v0.1.0-dictus 2>/dev/null || true
git ls-remote --tags origin | grep 'v0.1.0' || echo "ok — no conflicting remote tag"
```

    3. Click "Publish release"
