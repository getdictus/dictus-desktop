## Decisions Made

**1. Opaque-navy-tile variant for ICON-01 (approved by Pierre as Option A)**
The plan's original intent was a 1024×1024 RGBA PNG with transparent corners. Pierre instead created an opaque navy background tile using the GitHub avatar SVG rasterized at high DPI (`magick -background none -density 300 github-avatar.svg -resize 1024x1024`). This results in a solid navy square with no alpha channel — meaning the Linux black-corners artifact is physically impossible since there are no transparent pixels to render incorrectly. Pierre explicitly approved this approach and stated it achieves the same goal via absence-of-transparency rather than correct-transparency.

**2. ImageMagick fallback for 256×256 and 512×512**
The `tauri icon` CLI does not generate files named `256x256.png` or `512x512.png` (it uses `128x128@2x.png` for the 256px variant and `icon.png` for the 512px). Per plan §Pitfall 3 guidance, these were generated manually: `convert source.png -resize 256x256 src-tauri/icons/256x256.png` and similarly for 512x512. This is recorded explicitly so future maintainers know the provenance.

**3. Linux/Windows visual verification deferred**
Pierre cannot build Linux packages from macOS, and has no Windows machine available. Both platforms have automated backstops (see Deferred Visual Verification Risk). Pierre's statement: "si jamais les icônes vont pas, on corrigera ça plus tard et on fera des tests plus poussés plus tard, on va pas perdre de temps là-dessus, on va avancer."

**4. Mobile icons untouched**
`src-tauri/icons/android/` and `src-tauri/icons/ios/` directories were not modified — mobile assets are out of scope for this phase and require separate tooling.
