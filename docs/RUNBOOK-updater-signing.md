# Dictus Desktop — Updater Signing Runbook

> **This is the single canonical entry point for all Dictus Desktop releases.**
> Follow this document in order when setting up the signing key for the first time, rotating keys, or shipping a new release. Future-Pierre: this document exists so you do not have to re-research how the Tauri auto-updater works in six months.

---

## 1. Overview

Dictus Desktop uses the **Tauri v2 auto-updater** (`tauri-plugin-updater 2.10.0`) to deliver updates to users automatically. The update mechanism works as follows:

1. At each app launch, `UpdateChecker.tsx` invokes `trigger_update_check` (a Tauri command in `lib.rs`).
2. That command calls the updater plugin, which fetches `latest.json` from the configured endpoint: `https://github.com/getdictus/dictus-desktop/releases/latest/download/latest.json`.
3. `latest.json` is generated by `tauri-apps/tauri-action@v0` and contains the version number, per-platform download URLs, and **Ed25519 signatures** for each installer artifact.
4. The updater plugin verifies the signature of every downloaded artifact against the public key stored in `src-tauri/tauri.conf.json` (`plugins.updater.pubkey`). Verification uses **Minisign/Ed25519** — not OpenPGP, not RSA.
5. If verification passes, the installer is launched in-place and the app restarts at the new version.

Each installer artifact (`.dmg`, `.msi`, `.exe`, `.deb`, `.rpm`, `.AppImage`) is accompanied by a `.sig` file. These `.sig` files are produced during the build by `tauri-action` using the private key in the `TAURI_SIGNING_PRIVATE_KEY` GitHub Secret.

### Why YubiKey is NOT used as the signing backend

YubiKey was explicitly evaluated and rejected as the Tauri signing backend for three reasons:

1. **Tauri updater uses Minisign (Ed25519), not OpenPGP/SSH.** YubiKey supports Ed25519 via its OpenPGP interface, but that interface produces OpenPGP-formatted signatures. `bunx tauri signer` (the Minisign tool) cannot consume OpenPGP-format signatures — the formats are incompatible despite sharing the same underlying elliptic curve.
2. **The CI runner has no physical access to the YubiKey.** Signing happens inside GitHub Actions on every release run. The seven-platform build matrix runs on GitHub-hosted runners, which have no USB access. The alternatives (self-hosted runner on Pierre's Mac, or a cloud HSM) are disproportionate scope for a solo-maintained open source project.
3. **Neither `tauri signer` nor `tauri-apps/tauri-action@v0` support HSM or YubiKey backends as of 2026.** Building a custom Minisign-over-YubiKey wrapper would require forking or wrapping the signing toolchain, which is out of scope for this phase.

The YubiKey IS used in this project for its appropriate strengths: **encrypting the offline backup** of the private key file (via `age -R <yubikey_age_pubkey>`), GitHub 2FA, and optionally Git commit signing.

## 2. Key Custody

The Ed25519 private key must be stored in all three of the following locations before the local file is deleted. Loss of the private key after the first signed release ships is an **unrecoverable event** — see §4.

### Storage Locations (in priority order)

1. **GitHub Secrets (primary CI)** on `getdictus/dictus-desktop`:
   - `TAURI_SIGNING_PRIVATE_KEY` ← paste the full raw contents of `~/.tauri/dictus.key` (the entire file, including header/footer lines if present)
   - `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` ← the passphrase you set during generation
   - These secret names are **already referenced** in `.github/workflows/build.yml` at lines 354-355 — no workflow change is needed once the secrets exist in the repo settings UI.
   - Set secrets at: https://github.com/getdictus/dictus-desktop/settings/secrets/actions

2. **Bitwarden (primary backup)** — folder `getdictus`, two separate Secure Notes:
   - Note named **"Dictus Updater Signing Key"** → full contents of `~/.tauri/dictus.key`
   - Note named **"Dictus Updater Signing Passphrase"** → the passphrase only
   - **DO NOT store key material and passphrase in the same note.** This is defense in depth against partial vault compromise — if one note is somehow exposed, an attacker still cannot use the key without the passphrase.

3. **Offline backup (redundancy)** — `age`-encrypted file on an external drive or USB stick:
   ```bash
   age -R <Pierre's YubiKey age pubkey> ~/.tauri/dictus.key > dictus.key.age
   # Move dictus.key.age to external drive / USB
   ```
   This is the designated use of Pierre's YubiKey pair in this project. The age encryption allows decrypting the offline backup using the YubiKey without requiring online access to any service.

4. **Local disk** — `~/.tauri/dictus.key`:
   - **MUST be deleted** (`rm ~/.tauri/dictus.key`) **ONLY after** all three locations above are confirmed working.
   - The **public key** (`~/.tauri/dictus.key.pub`) is not sensitive — it is committed to `src-tauri/tauri.conf.json` as `plugins.updater.pubkey` and stays there.

## 3. First-Time Keypair Generation

Run this **once** on Pierre's Mac. Never run on a CI runner or shared machine.

```bash
# 1. Generate keypair with a strong passphrase
#    (choose a passphrase DISTINCT from your Bitwarden master password)
bunx tauri signer generate -w ~/.tauri/dictus.key

# 2. Read the public key to paste into tauri.conf.json
cat ~/.tauri/dictus.key.pub
# The output is a RAW BASE64 STRING (no PEM armor, no -----BEGIN/END headers).
# Paste this ENTIRE string as-is into plugins.updater.pubkey in src-tauri/tauri.conf.json.
# Do NOT paste the .key file (private key) — only the .key.pub (public key).

# 3. Back up the private key to Bitwarden note "Dictus Updater Signing Key"
cat ~/.tauri/dictus.key
# Copy the full output into the Bitwarden Secure Note (folder: getdictus)

# 4. Back up the passphrase to Bitwarden note "Dictus Updater Signing Passphrase"
#    (type the passphrase directly into the Bitwarden note — do not copy from terminal history)

# 5. Create the offline age-encrypted backup
age -R <yubikey_age_pubkey> ~/.tauri/dictus.key > dictus.key.age
# Move dictus.key.age to your external drive / USB stick immediately after

# 6. Set the GitHub Secrets in the repository settings UI
#    Navigate to: https://github.com/getdictus/dictus-desktop/settings/secrets/actions
#    Create two repository secrets:
#      - TAURI_SIGNING_PRIVATE_KEY          (paste full contents of ~/.tauri/dictus.key)
#      - TAURI_SIGNING_PRIVATE_KEY_PASSWORD (paste passphrase)
#    Verify: gh secret list --repo getdictus/dictus-desktop
#    (values are not shown, only names — that is expected)

# 7. Confirm all three backups are in place, then delete the local private key
rm ~/.tauri/dictus.key
# The .key.pub file can be kept or deleted — it is public and already in tauri.conf.json
```

**Security rules:**
- **Never commit** `~/.tauri/dictus.key` to any repository.
- **Never paste** the private key into Slack, email, GitHub Issues, PR comments, or any CI log.
- The **public key** (`~/.tauri/dictus.key.pub`) is safe to commit and is not sensitive.
- If you accidentally leak the private key, treat it as compromised and rotate immediately (see §5).

## 4. Recovery Plan (If Private Key Is Lost)

> **CRITICAL:** This is an unrecoverable risk that was accepted when designing this phase. Read carefully.

If the Ed25519 private key is lost **after the first signed release ships**, every installed copy of Dictus Desktop is **permanently stranded** on its current version. The Tauri updater will attempt to verify update artifacts against the old public key in `tauri.conf.json`, fail signature verification for any artifact signed with a new key, and report an error. There is no server-side mechanism to push a new public key to existing installs.

**Manual reinstall fallback for users:** `UpdateChecker.tsx` opens `https://github.com/getdictus/dictus-desktop/releases/latest` in the browser when the updater flow fails. This path works independently of the Ed25519 key and is the fallback for all users. Stranded users can always manually download the latest installer from GitHub Releases.

**Recovery procedure (if loss occurs):**
1. Regenerate a new keypair (§3).
2. Update `plugins.updater.pubkey` in `src-tauri/tauri.conf.json` with the NEW raw base64 pubkey.
3. Update `TAURI_SIGNING_PRIVATE_KEY` and `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` in GitHub Secrets.
4. Ship a new signed release (§6). All existing installs must manually reinstall this release once to get back into the update chain.
5. Communicate the situation to users via GitHub Releases notes and the project website.

**No in-app rotation mechanism is built in v1.1.** The "broken updater notification UI" (a non-dismissible banner surfacing repeated update-check failures) is explicitly deferred to post-v1.1. The manual reinstall fallback via `UpdateChecker.tsx` is the only safety net.

## 5. Key Rotation

Rotation is needed if: the private key is suspected compromised, or as a proactive security policy after a defined period.

**Rotation is NOT transparent to existing users.** Between the old and new signed releases, any user whose auto-update runs during the transition window will receive an artifact signed by the NEW key but has the OLD pubkey in `tauri.conf.json` — signature verification will fail. Ship the rotation release and follow-up releases quickly to minimize the window.

**Steps:**

1. Follow §3 to generate a new keypair (`bunx tauri signer generate -w ~/.tauri/dictus-new.key`).
2. Archive the old Bitwarden entries (rename to "Dictus Updater Signing Key (retired YYYY-MM-DD)" and "Dictus Updater Signing Passphrase (retired YYYY-MM-DD)"). **DO NOT delete** — keep for a transition period in case an old release needs to be diagnosed or re-verified.
3. Update `plugins.updater.pubkey` in `src-tauri/tauri.conf.json` with the NEW raw base64 pubkey from `~/.tauri/dictus-new.key.pub`.
4. Update GitHub Secrets `TAURI_SIGNING_PRIVATE_KEY` and `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` with the new key material.
5. Confirm all three backup locations (Bitwarden, age offline, GitHub Secrets) have the NEW key.
6. Delete the local private key file.
7. Ship a new release immediately via §6 to minimize the key-transition window.

## 6. Release Procedure

> **This is the canonical entry point for shipping any Dictus Desktop release.** Execute every step in order.

### 6.1 Pre-flight

```bash
# Clean up stale local tags from old test runs
# (Old test runs created these tags; release.yml will fail at tag creation if they exist locally)
git tag -d 0.1.0 v0.1.0-dictus 2>/dev/null || true

# Verify no conflicting remote tags exist
git ls-remote --tags origin | grep 'v0.1.0' && echo "REMOTE TAG CONFLICT — abort and investigate" || echo "ok — no remote tag conflict"
```

### 6.2 Version consistency

Starting from the SECOND release (after v0.1.0 ships), **any version bump MUST update all three version files in the same commit**:

```bash
# Assert version consistency before triggering release
grep '"version"' src-tauri/tauri.conf.json package.json
grep '^version' src-tauri/Cargo.toml
# All three version strings MUST match before proceeding
```

**Note for v0.1.0 initial release:** `package.json` currently shows `"version": "0.8.2"` (the upstream Handy value). Do NOT change it in this phase — the upstream package.json cleanup (TECH-03-adjacent) is deferred. The authoritative source for the release workflow is `src-tauri/tauri.conf.json`, which is what `release.yml:21` reads to determine the release tag.

### 6.3 Trigger the release

1. Navigate to: https://github.com/getdictus/dictus-desktop/actions/workflows/release.yml
2. Click **"Run workflow"** → select branch `main` → click **"Run workflow"**.
3. Wait for the **7-platform build matrix** to complete:
   - `macos-26` (Apple Silicon)
   - `macos-latest` (Apple Silicon / x86_64)
   - `ubuntu-22.04` (x86_64)
   - `ubuntu-24.04` (x86_64)
   - `ubuntu-24.04-arm` (aarch64)
   - `windows-latest` (x86_64)
   - `windows-11-arm` (aarch64)
4. A **draft release** named `v<version>` is automatically created with all platform artifacts attached.

### 6.4 Verify draft before publishing

Before publishing the draft release, verify all of the following at https://github.com/getdictus/dictus-desktop/releases:

- Artifacts are named with `dictus_*` prefix (e.g., `dictus_0.1.0_aarch64.dmg`). If you see `handy_*`, the asset-prefix CI fixes from Plan 02 were not applied — do NOT publish, investigate.
- `latest.json` is attached to the release.
- `.sig` files exist alongside each installer (`.dmg`, `.msi`, `.exe`, `.deb`, `.rpm`, `.AppImage`). Missing `.sig` files mean `bundle.createUpdaterArtifacts` is not `true` in `tauri.conf.json` — do NOT publish, investigate.

### 6.5 Publish

1. Click **"Edit"** on the draft release.
2. If "This is a pre-release" is checked and this is a real release, uncheck it.
3. Review the auto-generated release notes.
4. Click **"Publish release"**.

### 6.6 Post-release assertion (UPDT-10)

After publishing, run the verification script to confirm `latest.json` is live and structurally valid:

```bash
bash scripts/verify-updater-release.sh
```

Must exit 0. A non-zero exit (404, empty JSON, malformed response, missing platforms) means the release is broken — **unpublish the release immediately**, diagnose the issue, fix it, and re-release. Do NOT leave a broken `latest.json` endpoint live — existing installs will start failing their update checks at every launch.

## 7. Common Pitfalls

Cross-reference: `.planning/research/PITFALLS.md` §"Critical Pitfalls — Auto-Updater (v1.1)"

| Code | Summary |
|------|---------|
| A1 | **Key loss post-release** — private key lost after first signed release ships → all installs permanently stranded. See §4. |
| A2 | **Wrong asset-prefix** — release still produces `handy_*` artifacts if `release.yml:77` or `build.yml:22` is not updated to `dictus`. Both files must be changed. |
| A3 | **Wrong config section for `createUpdaterArtifacts`** — must be in `bundle`, NOT in `plugins.updater`. Setting it in the wrong section silently has no effect; `.sig` files will not be generated. |
| A4 | **PEM-wrapped pubkey** — `plugins.updater.pubkey` must be the raw base64 string from `.key.pub`, NOT a PEM-wrapped string. Paste the contents of the `.key.pub` file as-is. PEM-wrapped pubkey causes `UnexpectedKeyId` runtime error. |
| A5 | **Endpoint 404 before first publish** — once `endpoints` is populated in `tauri.conf.json`, the app starts checking. The URL 404s until the first release is published. This is expected during development. |
| A6 | **Stale local tags** — `git tag -d 0.1.0 v0.1.0-dictus` pre-flight in §6.1 prevents `release.yml` tag-creation failures. |
